# Understanding Big-O Notation

Big-O notation is the language we use to describe algorithmic complexity. It tells us how an algorithm's runtime or memory usage grows as the input size increases.

## The Core Idea

Imagine you have a list of `n` items. How does your algorithm behave as `n` grows?

- **O(1)** — Constant time. Doesn't matter if `n` is 10 or 10 million.
- **O(log n)** — Logarithmic. Doubles `n`, adds one step. Binary search.
- **O(n)** — Linear. Steps grow proportionally to input size.
- **O(n log n)** — Common in sorting algorithms like merge sort.
- **O(n²)** — Quadratic. Nested loops. Gets slow fast.
- **O(2ⁿ)** — Exponential. Only works for tiny inputs.

## A Concrete Example

```python
# O(n²) — check every pair
def has_duplicates_slow(arr):
    for i in range(len(arr)):
        for j in range(i + 1, len(arr)):
            if arr[i] == arr[j]:
                return True
    return False

# O(n) — use a hash set
def has_duplicates_fast(arr):
    seen = set()
    for item in arr:
        if item in seen:
            return True
        seen.add(item)
    return False
```

For `n = 10,000`:
- Slow version: ~50 million comparisons
- Fast version: ~10,000 lookups

## Space vs Time

Big-O applies to **both** time and memory. The fast version above is `O(n)` time but also `O(n)` space because of the `seen` set. Sometimes you trade one for the other.

## Practical Tips

1. **Focus on the dominant term**: `O(n² + n)` is just `O(n²)`.
2. **Drop constants**: `O(2n)` is `O(n)`.
3. **Know your data structures**: Hash maps give O(1) average lookup.
4. **Profile before optimizing**: Don't guess where the bottleneck is.

Understanding Big-O turns you from a programmer into an engineer. It's the foundation that everything else builds on.
