## 1. 서버-클라이언트 아키텍처의 본질

### 1.1 단순 정의를 넘어서

강의에서 서버는 "데이터나 서비스를 제공하는 컴퓨터"로 정의됩니다.  
하지만 실무에서는 이보다 훨씬 복잡한 개념입니다.

```
클라이언트 ──[Request]──▶ 로드 밸런서 ──▶ 웹 서버 (EC2)
                                              │
                                              ▼
                                         앱 서버 (EC2)
                                              │
                                              ▼
                                         DB 서버 (RDS)
클라이언트 ◀──[Response]──────────────────────┘
```

현대 웹 서비스는 단일 서버가 아닌 **다계층(Multi-Tier) 아키텍처**를 구성합니다:

### 1.2 프로토콜별 서버 유형과 포트

서버-클라이언트 통신은 **프로토콜(Protocol)** 로 정의됩니다. 각 서비스는 고유한 포트를 사용합니다.

```bash
# 주요 프로토콜과 포트 번호
HTTP    → 80/TCP    (웹 서버)
HTTPS   → 443/TCP   (SSL/TLS 암호화 웹 서버)
SSH     → 22/TCP    (보안 원격 접속)
SMTP    → 25/TCP    (메일 전송)
POP3    → 110/TCP   (메일 수신)
IMAP    → 143/TCP   (메일 동기화)
MySQL   → 3306/TCP  (데이터베이스)
```

> **실무 포인트**: EC2 Security Group 설정 시 필요한 포트만 허용하는 **최소 권한 원칙(Principle of Least Privilege)** 을 따르세요. 모든 포트를 0.0.0.0/0으로 열어두는 것은 심각한 보안 위협입니다.

### 1.3 웹 서버가 처리하는 데이터 심화

강의 자료의 표를 실무 관점으로 확장합니다:

```
웹 서버 처리 데이터 분류
├── 정적(Static) 콘텐츠
│   ├── HTML     → 구조 정의
│   ├── CSS      → 스타일링
│   ├── JS       → 클라이언트 사이드 로직
│   └── 미디어   → 이미지, 영상, 폰트
└── 동적(Dynamic) 콘텐츠
    ├── 서버 사이드 렌더링 (PHP, Python, Node.js)
    ├── API 응답 (JSON/XML)
    └── 데이터베이스 쿼리 결과
```

**AWS에서의 최적화 전략**:
- 정적 콘텐츠 → **S3 + CloudFront** (EC2 부하 감소, 비용 절감)
- 동적 콘텐츠 → **EC2 or Lambda** (비즈니스 로직 처리)

## 2. 서버 OS 선택 전략

### 2.1 Linux vs Windows Server: 심화 비교

```
Linux 선택 권장 시나리오
├── 웹 서버 구축 (Nginx, Apache)
├── 컨테이너/쿠버네티스 환경
├── 비용 최적화가 중요한 경우
├── DevOps/CI-CD 파이프라인
└── 오픈소스 기반 애플리케이션

Windows Server 선택 권장 시나리오
├── .NET Framework 기반 레거시 앱
├── Active Directory 통합 필요
├── IIS 기반 ASP.NET 애플리케이션
├── SQL Server + 윈도우 통합 인증
└── 기업 내부 워크로드 마이그레이션
```

### 2.2 Amazon Linux 2023: AWS 최적화 배포판

AWS는 자체 리눅스 배포판인 **Amazon Linux**를 제공합니다. 

```bash
# Amazon Linux 2023 주요 특징
- AWS 서비스와 최적화된 통합
- 5년 장기 지원 (LTS)
- SELinux 기본 활성화
- systemd 기반 서비스 관리
- AWS CLI, SSM Agent 사전 설치

# 패키지 관리 (dnf 사용)
sudo dnf update -y
sudo dnf install nginx -y
sudo systemctl enable --now nginx
```

### 2.3 리눅스 배포판별 패키지 관리 명령어

```bash
# Ubuntu/Debian (deb 기반)
sudo apt update && sudo apt upgrade -y
sudo apt install nginx mysql-server -y

# RHEL/CentOS/Amazon Linux (rpm 기반)
sudo yum update -y  # CentOS 7 이하
sudo dnf update -y  # RHEL 8+, Amazon Linux 2023

# 서비스 관리 (공통 - systemd)
sudo systemctl start nginx
sudo systemctl enable nginx   # 부팅 시 자동 시작
sudo systemctl status nginx
```

### 3.1 가상화 유형 심화

강의에서는 기본 가상화 개념만 다루지만, 실무에서는 세 가지 유형을 구분해야 합니다:

```
가상화 유형 비교
┌─────────────────────────────────────────────────────┐
│  Type 1 Hypervisor (Bare Metal)                     │
│  ┌──────┐ ┌──────┐ ┌──────┐                         │
│  │ VM1  │ │ VM2  │ │ VM3  │  ← Guest OS             │
│  └──────┘ └──────┘ └──────┘                         │
│  ┌─────────────────────────┐                         │
│  │      Hypervisor         │  (VMware ESXi, Xen)    │
│  └─────────────────────────┘                         │
│  ┌─────────────────────────┐                         │
│  │      Hardware           │                         │
│  └─────────────────────────┘                         │
│  예: AWS EC2 (Nitro Hypervisor)                      │
└─────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────┐
│  Container (컨테이너 가상화)                          │
│  ┌────────┐ ┌────────┐ ┌────────┐                   │
│  │Container│ │Container│ │Container│ ← App + Libs   │
│  └────────┘ └────────┘ └────────┘                   │
│  ┌─────────────────────────┐                         │
│  │    Container Runtime    │  (Docker, containerd)  │
│  └─────────────────────────┘                         │
│  ┌─────────────────────────┐                         │
│  │      Host OS + Kernel   │                         │
│  └─────────────────────────┘                         │
│  예: AWS ECS, EKS                                    │
└─────────────────────────────────────────────────────┘
```

### 3.2 AWS Nitro System: EC2의 핵심 기술

AWS EC2는 자체 개발한 **Nitro System** 기반으로 운영됩니다:

```
AWS Nitro System 구성요소
├── Nitro Hypervisor
│   └── 경량화된 하이퍼바이저 → 더 많은 자원을 인스턴스에 할당
├── Nitro Cards
│   ├── Storage Card (EBS 오프로딩)
│   ├── Network Card (VPC 네트워킹 오프로딩)
│   └── Monitoring Card (하드웨어 모니터링)
└── Nitro Security Chip
    └── 하드웨어 수준 보안 격리
```

**핵심 장점**: 기존 하이퍼바이저 오버헤드를 하드웨어로 오프로딩하여 **베어메탈에 근접한 성능** 제공

## 4. Amazon EC2 완전 해부

### 4.1 인스턴스 유형 명명 규칙 심화

강의 자료의 `m6g.medium` 예시를 더 깊이 이해합니다:

```
인스턴스 유형 명명 규칙 완전 해석

  c  7  g  n  .  2xlarge
  │  │  │  │     │
  │  │  │  │     └── 크기 (nano < micro < small < medium < large
  │  │  │  │                < xlarge < 2xlarge < ... < 48xlarge)
  │  │  │  └────── 추가 속성
  │  │  │          a = AMD EPYC CPU
  │  │  │          g = AWS Graviton (ARM) CPU
  │  │  │          i = Intel CPU 명시
  │  │  │          n = 강화된 네트워킹 (25/100Gbps)
  │  │  │          d = 로컬 NVMe SSD 포함
  │  │  │          b = EBS 최적화
  │  │  │          e = 확장 메모리/스토리지
  │  │  └───────── 세대 (숫자가 클수록 최신, 보통 더 빠르고 저렴)
  │  └──────────── 인스턴스 패밀리 (세대 번호)
  └─────────────── 인스턴스 패밀리
                   c = Compute Optimized (컴퓨팅 집약)
                   m = General Purpose (범용)
                   r = Memory Optimized (메모리 집약)
                   t = Burstable (버스트 가능 범용)
                   p = GPU (ML 추론/학습)
                   g = GPU (그래픽/ML)
                   i = Storage Optimized (스토리지 I/O)
                   d = Dense Storage (대용량 스토리지)
                   f = FPGA
                   x = Extra Memory
                   z = High Frequency (고주파 CPU)
```

### 4.2 인스턴스 패밀리별 실무 사용 사례

```
워크로드별 최적 인스턴스 선택 가이드

웹 서버 / 소규모 앱
└── t3.micro ~ t3.medium (버스트 가능, 비용 효율)

API 서버 / 마이크로서비스
└── c7g.large ~ c7g.2xlarge (CPU 집약, Graviton 성능/비용 우수)

In-Memory 데이터베이스 (Redis, Memcached)
└── r7g.large ~ r7g.4xlarge (높은 메모리 대 CPU 비율)

ML 학습 / GPU 컴퓨팅
└── p3.2xlarge (NVIDIA V100), p4d.24xlarge (A100 x8)

분석 쿼리 / Hadoop
└── i3en.large (로컬 NVMe SSD, 고 I/O 처리량)

고성능 컴퓨팅 (HPC)
└── hpc7g.16xlarge (EFA 네트워크, 초저지연)
```

### 4.3 T 계열 인스턴스: CPU Credit 메커니즘

`t2.micro`, `t3.micro` 등 버스터블 인스턴스의 핵심 개념:

```
CPU Credit 동작 원리

일반 상태 (CPU 사용률 < 기준선)
└── 크레딧 적립 (시간당 일정량)

버스트 상태 (CPU 사용률 > 기준선)
└── 크레딧 소진

크레딧 소진 시
├── t2: CPU 기준선으로 성능 제한
└── t3: unlimited 모드 → 추가 비용 발생 (주의!)

인스턴스별 기준선 CPU 및 크레딧 적립률
┌──────────────┬──────────┬──────────────────┐
│ 인스턴스 유형 │ vCPU 수  │ 기준선 CPU (%)   │
├──────────────┼──────────┼──────────────────┤
│ t3.nano      │ 2        │ 5%               │
│ t3.micro     │ 2        │ 10%              │
│ t3.small     │ 2        │ 20%              │
│ t3.medium    │ 2        │ 20%              │
│ t3.large     │ 2        │ 30%              │
└──────────────┴──────────┴──────────────────┘
```

> **⚠️ 실무 주의사항**: 프리 티어 `t2.micro`는 `t2` 계열로 크레딧 소진 시 성능이 급격히 저하됩니다.  
> 운영 환경에서는 `t3` 이상, 또는 고정 성능이 필요하다면 `m` 계열을 선택하세요.

### 4.4 인스턴스 상태 전이 심화: 요금 발생 구간

강의에서 다룬 7가지 상태에 **요금 발생 여부**를 추가합니다:

```
인스턴스 상태별 요금 발생 현황

상태              EC2 요금    EBS 요금    EIP 요금
─────────────────────────────────────────────────
대기 중(pending)    ✅ 발생     ✅ 발생     ✅ 발생
실행 중(running)    ✅ 발생     ✅ 발생     ✅ 발생
중지 중(stopping)   ✅ 발생     ✅ 발생     ✅ 발생
중지됨(stopped)     ❌ 없음     ✅ 발생     ✅ 발생  ← 주의!
재부팅(rebooting)   ✅ 발생     ✅ 발생     ✅ 발생
종료 중(shutting)   ✅ 발생     ✅ 발생     ✅ 발생
종료됨(terminated)  ❌ 없음     ❌ 없음     ❌ 없음
```

> **💡 비용 최적화 포인트**:  
> - 인스턴스를 **중지(Stop)** 해도 EBS 볼륨과 Elastic IP 요금은 계속 발생합니다.  
> - 테스트 후 비용 절감을 위해서는 **종료(Terminate)** + 필요 시 스냅샷 보관이 효율적입니다.

## 5. EC2 스토리지 심층 비교

### 5.1 스토리지 유형 완전 비교

```
AWS EC2 스토리지 생태계

├── 블록 스토리지 (Block Storage)
│   ├── 인스턴스 스토어 (Ephemeral)
│   │   ├── 물리 서버에 직접 연결
│   │   ├── 최고 I/O 성능 (수백만 IOPS)
│   │   └── 인스턴스 종료 시 데이터 소실
│   └── Amazon EBS (Elastic Block Store)
│       ├── 네트워크 연결 블록 스토리지
│       ├── 인스턴스 종료 후에도 데이터 유지
│       └── 볼륨 타입:
│           ├── gp3 (범용 SSD) - 권장
│           ├── io2 Block Express (고성능 IOPS)
│           ├── st1 (처리량 최적화 HDD)
│           └── sc1 (저빈도 액세스 HDD, 최저가)
│
├── 파일 스토리지 (File Storage)
│   └── Amazon EFS (Elastic File System)
│       ├── 여러 EC2 인스턴스 동시 마운트
│       ├── NFS v4 프로토콜
│       └── 자동 용량 조절
│
└── 객체 스토리지 (Object Storage)
    └── Amazon S3
        ├── 정적 파일, 백업, 로그 저장
        ├── EC2에서 SDK/CLI로 접근
        └── 무제한 용량, 99.999999999% 내구성
```

### 5.2 EBS 볼륨 타입별 성능 비교표

| 볼륨 타입 | 용도 | 최대 IOPS | 최대 처리량 | 비용 |
|-----------|------|-----------|------------|------|
| gp3 | 범용 (권장) | 16,000 | 1,000 MB/s | $$ |
| gp2 | 범용 (구형) | 16,000 | 250 MB/s | $$ |
| io2 Block Express | DB, 미션 크리티컬 | 256,000 | 4,000 MB/s | $$$$ |
| io1 | 고성능 DB | 64,000 | 1,000 MB/s | $$$ |
| st1 | 빅데이터, 로그 | 500 | 500 MB/s | $ |
| sc1 | 콜드 스토리지 | 250 | 250 MB/s | $ (최저) |

### 5.3 EBS 스냅샷과 데이터 보호 전략

```bash
# AWS CLI로 EBS 스냅샷 생성
aws ec2 create-snapshot \
  --volume-id vol-0123456789abcdef0 \
  --description "Production DB backup - $(date +%Y%m%d)"

# 스냅샷에서 볼륨 복원
aws ec2 create-volume \
  --snapshot-id snap-0123456789abcdef0 \
  --availability-zone ap-northeast-2a \
  --volume-type gp3

# 자동 스냅샷: AWS Data Lifecycle Manager (DLM) 활용 권장
# → 규칙 기반 자동화, 보존 정책, 교차 리전 복제 지원
```

### 6.2 퍼블릭 IP vs Elastic IP vs 프라이빗 IP

```
IP 주소 유형 비교

┌────────────────┬──────────────────────────────────────────┐
│ 유형           │ 특징                                      │
├────────────────┼──────────────────────────────────────────┤
│ 프라이빗 IP    │ • VPC 내부 통신 전용                      │
│                │ • 인스턴스 재시작 후에도 유지              │
│                │ • 인터넷에서 직접 접근 불가                │
│                │ • RFC 1918 대역 (10.x, 172.16-31.x,      │
│                │   192.168.x)                              │
├────────────────┼──────────────────────────────────────────┤
│ 퍼블릭 IP      │ • 인터넷과 통신 가능                      │
│ (자동 할당)    │ • 인스턴스 중지/시작 시 변경됨            │
│                │ • 무료                                    │
├────────────────┼──────────────────────────────────────────┤
│ Elastic IP     │ • 고정 퍼블릭 IP                          │
│ (EIP)          │ • 인스턴스 교체 시 재할당 가능            │
│                │ • 인스턴스에 연결 안 된 경우 요금 발생    │
│                │ • DNS 레코드 고정 시 사용                 │
└────────────────┴──────────────────────────────────────────┘
```

### 6.3 ENI 고급 활용 패턴

```bash
# ENI는 단순 네트워크 어댑터 이상의 활용이 가능합니다

# 사용 사례 1: 듀얼 네트워크 인터페이스
# → 퍼블릭 ENI + 프라이빗 ENI를 한 인스턴스에 연결
# → 서로 다른 서브넷/보안 그룹 적용

# 사용 사례 2: 장애 조치 (Failover)
# → 주 인스턴스 장애 시 ENI를 대기 인스턴스로 빠르게 이동
# → IP 주소 유지로 DNS 전파 지연 없이 전환

# 사용 사례 3: 라이선스 관리
# → MAC 주소 기반 라이선스를 ENI로 관리
# → ENI를 새 인스턴스에 이동하여 라이선스 유지
```

## 7. 보안 모델: Security Group과 키 페어

### 7.1 Security Group 심화: Stateful 방화벽

강의에서 "가상의 방화벽"으로 설명하지만, 핵심 특성은 **Stateful(상태 유지형)** 이라는 점입니다:

```
Stateful vs Stateless 방화벽 차이

Stateful (Security Group) - AWS 사용
┌─────────────────────────────────────────┐
│ 인바운드 규칙: 포트 443 허용             │
│                                         │
│ 클라이언트 → [443 요청] → EC2 ✅         │
│ 클라이언트 ← [응답 트래픽] ← EC2 ✅ 자동  │
│ (아웃바운드 규칙 없어도 응답 자동 허용)  │
└─────────────────────────────────────────┘

Stateless (Network ACL) - 서브넷 레벨
┌─────────────────────────────────────────┐
│ 인바운드 규칙: 포트 443 허용             │
│ 아웃바운드 규칙: 포트 1024-65535 허용   │
│ (양방향 모두 명시적으로 허용 필요)      │
└─────────────────────────────────────────┘
```

### 7.2 최소 권한 원칙 적용: 보안 그룹 설계

```yaml
# 실무 보안 그룹 설계 패턴
# (CloudFormation 예시)

WebTierSG:
  Type: AWS::EC2::SecurityGroup
  Properties:
    GroupDescription: "Web Tier - Public Facing"
    VpcId: !Ref VPC
    SecurityGroupIngress:
      - IpProtocol: tcp
        FromPort: 443
        ToPort: 443
        CidrIp: 0.0.0.0/0       # HTTPS 전체 허용
      - IpProtocol: tcp
        FromPort: 80
        ToPort: 80
        CidrIp: 0.0.0.0/0       # HTTP (HTTPS 리다이렉트용)
    SecurityGroupEgress:
      - IpProtocol: tcp
        FromPort: 8080
        ToPort: 8080
        DestinationSecurityGroupId: !Ref AppTierSG  # 앱 계층으로만

AppTierSG:
  Type: AWS::EC2::SecurityGroup
  Properties:
    GroupDescription: "Application Tier - Internal Only"
    SecurityGroupIngress:
      - IpProtocol: tcp
        FromPort: 8080
        ToPort: 8080
        SourceSecurityGroupId: !Ref WebTierSG   # 웹 계층에서만!
    SecurityGroupEgress:
      - IpProtocol: tcp
        FromPort: 3306
        ToPort: 3306
        DestinationSecurityGroupId: !Ref DBTierSG   # DB로만
```

## 8. 모니터링과 자동 스케일링

### 8.1 CloudWatch 지표 이해

```
CloudWatch 기본 지표 (5분 간격, 무료)
├── CPU
│   └── CPUUtilization (%)
├── 네트워크
│   ├── NetworkIn (bytes)
│   └── NetworkOut (bytes)

→ CloudWatch Agent 설치 필요 (커스텀 지표)
```

### 8.2 Auto Scaling: 동적 확장 전략

강의의 CPU 기반 스케일링을 실무 수준으로 확장합니다:

```
Auto Scaling 정책 유형

1. 단순 스케일링 (Simple Scaling) - 기초
   CPUUtilization > 70% → 인스턴스 2개 추가 (5분 쿨다운)
   
2. 단계별 스케일링 (Step Scaling) - 권장
   CPUUtilization 60-70%  → 1개 추가
   CPUUtilization 70-85%  → 2개 추가
   CPUUtilization 85-100% → 4개 추가
   
3. 대상 추적 스케일링 (Target Tracking) - 가장 스마트
   "CPUUtilization을 항상 50%로 유지"
   → AWS가 자동으로 추가/제거 결정
   
4. 예약 스케일링 (Scheduled Scaling)
   매일 09:00 → 최소 10개로 증가
   매일 22:00 → 최소 2개로 감소
   (예측 가능한 트래픽 패턴에 효과적)
   
5. 예측 스케일링 (Predictive Scaling) - 최신
   머신러닝으로 트래픽 예측
   → 수요 발생 전 미리 스케일 아웃
```

## 10. 서버리스: Lambda와 이벤트 드리븐 아키텍처

### 10.1 Lambda vs EC2: 언제 무엇을 선택할까?

강의에서 Lambda의 기본 개념을 다루었습니다. 선택 기준을 심화합니다:

```
Lambda 적합 시나리오
✅ 이벤트 트리거 기반 처리
   (S3 업로드 → 이미지 리사이징)
✅ 간헐적, 예측 불가한 트래픽
✅ 최대 실행 시간 15분 이내
✅ 상태 비저장(Stateless) 처리
✅ 마이크로서비스 각 기능 구현

EC2 적합 시나리오
✅ 장시간 실행 프로세스 (15분 초과)
✅ 특정 OS/미들웨어 설정 필요
✅ 고정적인 고부하 워크로드
✅ 레거시 애플리케이션 리프트앤시프트
✅ 커스텀 네트워킹 설정 필요
```

### 10.2 Lambda 실행 모델 이해

```
Lambda Cold Start vs Warm Start

Cold Start (첫 실행 또는 오랜 유휴 후)
┌─────────────────────────────────────────────┐
│ 컨테이너 생성 → 런타임 초기화 → 함수 로드   │
│ → 핸들러 실행                               │
│ 지연 시간: 100ms ~ 수 초 (언어/크기별 상이) │
└─────────────────────────────────────────────┘

Warm Start (최근 실행된 함수)
┌─────────────────────────────────────────────┐
│ (컨테이너 재사용) → 핸들러 실행만           │
│ 지연 시간: 1ms ~ 수십 ms                    │
└─────────────────────────────────────────────┘

Cold Start 최소화 전략
├── Provisioned Concurrency (예약된 동시성)
├── 경량 런타임 선택 (Python > Java 초기화 속도)
├── 배포 패키지 최소화
└── /tmp 디렉토리 캐싱 활용
```

### 10.3 Lambda 이벤트 드리븐 아키텍처 패턴

```
실무 Lambda 아키텍처 패턴

1. 비동기 이미지 처리 파이프라인
   
   사용자 업로드 → S3 버킷 → [S3 이벤트]
                                    │
                                    ▼
                              Lambda 함수
                              (이미지 리사이징)
                                    │
                         ┌──────────┴──────────┐
                         ▼                     ▼
                   S3 (썸네일)           DynamoDB (메타데이터)

2. API Gateway + Lambda (서버리스 REST API)
   
   클라이언트 → API Gateway → Lambda → DynamoDB
                                     ↘ RDS
                                     ↘ 외부 API

3. 이벤트 기반 ETL (데이터 처리)
   
   Kinesis Data Stream → Lambda → S3/Redshift
   (실시간 로그 수집)    (변환)   (데이터 웨어하우스)
```

AWS 컴퓨팅 서비스 핵심 정리
